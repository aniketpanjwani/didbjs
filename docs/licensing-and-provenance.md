# Licensing And Provenance

Status: accepted for conformance profile v1.

This document records the license/provenance posture for the public package. It
is not legal advice. The root `PROVENANCE.md` file is the public ledger.

## License Decision

Decision: target a permissive R package license compatible with MIT-derived
material, while using a clean-room process for the GPL-3.0 Stata reference.

Rationale:

- Python `gmarinichev/did_imputation` and Kyle R `didimputation` are MIT-signaled
  references and may be inspected or adapted with attribution and license notice
  preservation.
- Stata `borusyak/did_imputation` is GPL-3.0 and must not be translated or
  adapted into non-GPL implementation code.
- Stata remains the canonical black-box behavior oracle for estimator semantics,
  diagnostics, and plotting extraction behavior.

Rejected alternative: GPL-3.0 package allowing direct adaptation of Stata code.
That remains available only through a new decision record and conformance
profile.

## Operational Clean-Room Rules

- Implementation agents may read behavior specs, help files, mathematical
  descriptions, generated expected outputs, and this conformance contract.
- Implementation agents must not translate Stata source code line-by line or copy
  Stata comments, variable names, control structure, or distinctive expression.
- Stata expected outputs are generated by scripts and consumed as black-box
  fixtures.
- Hand-solvable and algebraic fixtures are required for the first core estimator
  path so code-to-code agreement is not the only oracle.
- A pre-release provenance review must compare implementation files against
  adapted MIT material and the GPL boundary.

## Known Source Exposure Before Implementation

Planning agents have inspected the Stata repository, including `.ado` and help
files, to inventory behavior. This is allowed for the planning contract. Future
implementation work must rely on frozen specs and generated outputs rather than
source translation.

## MIT Attribution Obligations

If Python or Kyle code, tests, examples, or documentation are copied or adapted:

- preserve license text;
- add copyright notices;
- record files/fragments in the provenance ledger;
- add SPDX headers where appropriate;
- cite adapted fixtures and examples.

## Data And Fixture Rights

Committed package data must be limited to:

- synthetic data generated in this repository; or
- source data whose redistribution rights are recorded in the provenance ledger.

Generated scalar/vector/matrix outputs from Stata, Python, and Kyle may be used
as parity targets when the source input data rights are documented. Do not
redistribute saved X/Twitter JSON as package data.

## Stata License Use

Stata regeneration is restricted to a maintainer-controlled licensed Stata
environment. Public CI and default package tests consume committed expected
artifacts rather than invoking Stata.
